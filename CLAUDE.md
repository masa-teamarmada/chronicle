# CLAUDE.md — 共通ルール

> **CLAUDE.mdの2層構成ルール**
> - **このファイル（共通）**: 運用ルール・開発規約・設計ログ・引き継ぎルール等。全プロジェクト共通。
> - **各プロジェクトのCLAUDE.md**: そのプロジェクト固有の技術仕様（DB構成・API・キャラ設定等）のみ。運用ルールは書かない。
> - 矛盾がある場合は、この共通ファイルが優先。

## 共通技術ルール

### GAS共通パターン
- Google Apps Script + Google Spreadsheet が正本DB
- `google.script.run` はそのページを配信したGASプロジェクト内の関数しか呼べない。別プロジェクトの関数は参照不可
- ファイル命名: `{番号}_{機能名}.js` / `{番号}_{機能名}.html`
- HTML内の `<style>`, `<script>` タグ内に直接コードを書く（GAS HTMLサービスの制約）

### ログ出力形式
変更ログは以下のJSON形式で出力する：
```json
{ "file": "ファイル名", "function": "関数名", "change": "何をどう変えたか" }
```

### 回答の基本姿勢 — 本質志向
- **そのセッションで解決すべき本質的な課題を常に意識する**。目の前の症状への場当たり的な対処ではなく、依頼の根本にある問題を解きにいく
- **回答する前に必ず反芻する**：「この回答は、セッションの本来の目的に対して適切な解決策か？」を自問し、対症療法に逸れていないか確認してから回答する
- 部分的な修正や一時的な回避策を提示する場合は、それが本質的な解決ではないことを明示し、根本対応の方向性も併せて示す

### Webデザインルール
- HTML/CSSのUI・ページデザインを新規作成または大幅改修するときは、**必ずAI Designer MCP（`generate_design` / `refine_design`）を使う**。えいみが直接HTML/CSSを手書きしない
- 生成後のコード調整・既存コードへの組み込み・軽微な修正はえいみが直接行ってOK
- AI Designerを使った場合は、その旨をユーザーに伝える（「AI Designerで生成したよ」等）

### えいみができることは全部えいみがやる
- CLI操作（supabase deploy、SQL実行、clasp push等）、設定ファイル編集（pbxproj、config.toml等）、マイグレーション実行など、**えいみ側で実行可能な作業はすべてえいみが行う**
- ソフトウェアのインストール（winget install等）、環境変数の設定、APIキーの設定（既存のコードやプロジェクト内から取得して設定）もえいみが行う
- まさに「このSQLを実行して」「Xcodeでファイル追加して」等の手作業を振らない
- まさにお願いするのは、えいみが物理的にできないこと（実機テスト、App Store操作等）だけ

### コード変更の基本姿勢
- 事実と解釈の徹底分離 / append-only / 人の判断を消さない
- 既存コードの意図を理解してから変更する
- 安全側に倒す（データ消失を避ける）
- **手動実行関数には必ずファイル名を添える** — GASエディタから手動実行する関数を案内するときは `関数名`（ファイル名）の形式で書く。例: `temp_testReceiptFolder()`（999_TempFunctions.js）

## フィードバックループ

コードを変更したら、以下を実行して品質を確認する：
- GAS関数の場合: 変更後に `clasp push --force` でデプロイ可能か確認
- HTML/JSの場合: 構文エラーがないか確認
- 3ファイル以上の変更時: 影響範囲を一覧で示す

## コンテキスト管理

### Compaction時の保持事項
compactされた場合、以下を必ず保持する：
- 変更済みファイル一覧と各変更内容
- 現在のタスク進捗状況（TodoWriteの内容）
- 未解決のエラーや問題点
- ユーザーとの合意事項

### 大規模変更時のルール
- 3ファイル以上を変更する場合、Plan Modeで計画を先に示す
- 既存コードを読んでから変更する（推測で変更しない）
- 変更前後のdiffを簡潔に示す

## セッション引き継ぎ

### ハンドオフファイルの構成
- ファイル名: `HANDOFF_{テーマ名}.md`（例: `HANDOFF_monthly_report.md`）
- 配置先: `design_log/{project}/`
- **1テーマ = 1ファイル**。セッションIDではなく作業テーマで分ける
- 並行セッションは別テーマなら別ファイル。同テーマの続きなら同じファイルを上書き

### セッション開始時（必須）
1. `design_log/{project}/` 内の `HANDOFF_*.md` を一覧確認する（`ls`）
2. ユーザーに「どのテーマの続き？ or 新規？」を確認する
3. 該当するハンドオフを読み、未完了タスクと次のアクションを把握する
4. 必要に応じて直近の設計ログ（同月の `YYYY-MM_design_log.md`）も読む

### セッション終了時（必須）
ユーザーが「終わり」「また今度」「引っ越す」等でセッション終了を示したら、**必ず**該当テーマの `HANDOFF_{テーマ名}.md` を更新する：
- **最終更新**: 日時とセッション概要（1行）
- **完了タスク**: このセッションで完了したこと
- **未完了・継続タスク**: 残っている作業（具体的に）
- **既知の問題・ブロッカー**: ハマったこと、未解決エラー
- **次のアクション（推奨）**: 次セッションで最初にやるべきこと
- **このセッションで得た知見**: 次のえいみが知っておくべきこと

### ハンドオフのライフサイクル
- テーマの作業が完全に完了したら、ファイルを削除する（残骸を増やさない）
- セッションから複数テーマにフォークする場合は、新しいハンドオフファイルを作る

### compaction時
コンテキスト圧縮が発生したら、該当ハンドオフの内容を保持事項として扱う。

## 設計ログ運用

### 基本ルール
- 設計判断・方針変更・アーキテクチャ決定を行ったら `claude/design_log/` 配下のmdに追記する
- **正本はmdファイル**。スプシは閲覧ビュー（md→スプシの自動同期で生成）
- 追記時はTodoに「設計ログ追記」を含め、完了を明示する

### ディレクトリ構成とプロジェクト対応表
```
design_log/
  amd_os/       ← AMD OS関連
  chronicle/    ← Chronicle関連
  new_biz/      ← 新規事業（ウェブ+SNS）関連
  freee/        ← freee会計関連
```

| design_log | 開発フォルダ | GASプロジェクト | 概要 |
|---|---|---|---|
| `amd_os/` | `claude/AMD_OS/` | `amd-os/` | AMD OS本体 |
| `chronicle/` | `claude/Chronicle/` | — | Chronicle（羅針盤） |
| `new_biz/` | `claude/tsukuyomi-site/` | — | 新規事業（ウェブ+SNS収益化） |
| `freee/` | — | — | freee会計（個人アカウント経理） |

※ プロジェクト追加時はこの表も更新すること

### ファイル命名
- `YYYY-MM_{機能名}.md`（例: `2026-04_monthly_report_refactor.md`）
- 1機能 = 1ファイル。月をまたぐ場合は開始月で命名

### 記載フォーマット
```markdown
# {機能名}

## YYYY-MM-DD: {判断タイトル}
- **背景**: なぜこの判断が必要だったか
- **決定**: 何をどうすることにしたか
- **理由**: なぜその選択肢を選んだか
- **影響範囲**: 変更が及ぶファイル・機能
```
